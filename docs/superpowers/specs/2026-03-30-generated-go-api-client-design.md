# Generated Go API Client Design

## Goal

Add a checked-in generated Go client derived from the Huma OpenAPI document and use it as the standard client for integration-style backend tests.

This should eliminate hand-maintained request construction in server tests, keep backend tests aligned with the published API contract, and make the generated client reusable for any future integration-style test in the repository.

## Current State

- The backend API is defined with Huma in `internal/server/huma_routes.go`.
- OpenAPI is generated by `cmd/middleman-openapi` and checked in at `frontend/openapi/openapi.json`.
- The frontend TypeScript schema is generated from that spec in `frontend/src/lib/api/generated/schema.ts`.
- Backend integration tests in `internal/server/api_test.go` still construct raw `httptest` requests directly against the server.
- There is no generated Go client in the repository today.

## Requirements

- Keep OpenAPI generation reproducible and checked in.
- Generate a reusable Go client from the checked-in OpenAPI document.
- Prefer the generated Go client for integration-style backend tests.
- Keep the generation flow simple enough to run from `make api-generate` and from pre-commit.
- Mark generated artifacts as generated for repository tooling, including GitHub Linguist classification.
- Avoid runtime generation in tests.

## Approaches Considered

### 1. Checked-in generated Go client from the checked-in OpenAPI spec

Use Huma to generate the OpenAPI document, then use `oapi-codegen` to generate a Go client package committed to the repository.

Pros:

- Single contract source
- Deterministic local and CI behavior
- Reusable from any integration-style test
- Closely matches Huma’s documented guidance

Cons:

- Adds generated Go code to the repo
- Requires one more generation step in tooling

### 2. Generate the Go client dynamically during tests

Start the server in tests, fetch `/api/v1/openapi.json`, and generate a client on demand.

Pros:

- No checked-in generated Go code

Cons:

- Slow
- Fragile
- Poor fit for normal Go tooling
- Harder to review and cache

### 3. Handwritten Go test client

Write a small typed wrapper over `http.Client` manually.

Pros:

- Minimal tooling change

Cons:

- Reintroduces contract drift
- Duplicates API shape by hand
- Does not scale as endpoints evolve

## Decision

Use approach 1.

The repository will keep generating the OpenAPI document with Huma, then generate a checked-in Go client from that document with `oapi-codegen`. The generated Go client will become the default path for integration-style backend tests.

## Design

### OpenAPI and generation flow

`cmd/middleman-openapi` remains the canonical entrypoint for OpenAPI emission. It will produce the checked-in API description used by both frontend and Go client generation.

Because Huma’s client-generation guidance targets `oapi-codegen`, the emitted document should be compatible with that workflow. If needed, the generator should explicitly emit an OpenAPI 3.0.3 document for the Go client path while keeping the checked-in output stable and reviewable.

`make api-generate` will be expanded so one command regenerates:

- the checked-in OpenAPI document
- the generated frontend TypeScript schema
- the generated Go client

The same generation command will remain in pre-commit so checked-in generated artifacts stay synchronized.

### Go client package layout

Add a dedicated package for generated client code, separate from the server package. A concrete target such as `internal/apiclient/generated` is appropriate.

The generated package should be produced by `go generate` from a small checked-in driver file. That file should:

- declare the package location for the generated client
- hold the `//go:generate` command
- keep the invocation discoverable in normal Go tooling

Alongside the generated package, add a small handwritten wrapper package, such as `internal/apiclient`, for construction and test ergonomics. That wrapper should be responsible only for:

- creating the generated client against an `httptest.Server` base URL
- centralizing any default headers or auth setup if needed later
- hiding generator-specific awkwardness when a small stable helper improves readability

The wrapper must stay intentionally thin. API behavior remains owned by the generated client and the server contract.

### Test strategy

Integration-style backend tests should prefer the generated Go client instead of building raw HTTP requests directly.

`internal/server/api_test.go` will be reworked to:

- start the server under test
- build the Go API client against the test server URL
- exercise the documented operations through generated methods
- assert on typed responses and expected HTTP status handling

Direct `httptest` requests should remain only where the test is specifically about low-level HTTP behavior that a client intentionally hides, such as:

- serving the raw OpenAPI document
- SPA fallback behavior
- non-API transport edge cases

This keeps contract tests aligned with the published API while preserving direct coverage where transport details matter.

### Generated artifact labeling

Generated artifacts must be clearly marked as generated for repository tooling.

Use `.gitattributes` to mark generated paths with `linguist-generated=true`. This should include:

- the generated Go client path
- the generated frontend schema path
- any other machine-generated API artifacts that should not dominate code review language statistics

Comments at the top of generated entrypoint files should also make the generated nature explicit where practical, but `.gitattributes` is the authoritative mechanism for Linguist labeling.

### Documentation updates

Update `CLAUDE.md` to document:

- Huma as the web framework
- OpenAPI-generated clients as the preferred integration-test path
- `go generate` and `make api-generate` as the supported regeneration mechanisms

## Verification

Implementation should be considered complete only if all of the following pass:

- `make api-generate`
- `make lint`
- `go test ./...`

Coverage should be compared before and after the server test migration to ensure the generated-client approach does not reduce backend coverage.

## Risks and mitigations

### Risk: `oapi-codegen` compatibility with emitted OpenAPI version

Mitigation:

- follow Huma’s documented client-generation path
- pin the generation flow explicitly in repo tooling
- if necessary, emit a Go-client-friendly OpenAPI variant in a deterministic way

### Risk: generated client method names are awkward in tests

Mitigation:

- keep a thin handwritten wrapper for test ergonomics
- avoid hiding real API behavior in that wrapper

### Risk: generated code churn increases review noise

Mitigation:

- keep generated output checked in and reproducible
- mark generated paths with `linguist-generated=true`
- centralize regeneration behind one command and pre-commit automation

## Scope boundaries

This design covers only:

- OpenAPI-to-Go-client generation
- backend integration-test migration to the generated client
- associated tooling and documentation updates

It does not include:

- replacing the frontend runtime client in this step
- changing API semantics
- broader database or service-layer refactors
